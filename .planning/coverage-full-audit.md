# 전 코퍼스 풀 커버리지 감사 — chunker 1.3.4

> 풀 파싱 + 풀 분석 (샘플링 없음). 21개 인덱싱 대상 파일 전부.
>
> 캐시: `.planning/audit-cache/<stem>.json` (PDF 1개, HWP 17개, HWPML 미파싱 3개)
> 분석: `.planning/audit-cache/analysis/<stem>.json`
> 분석 코드: `.planning/audit-cache/parse_all.py`, `.planning/audit-cache/analyze.py`

## 1. Executive Summary

**chunker 1.3.4 는 매뉴얼 PDF 1개 파일에서 명확한 회귀를 발생시킵니다. 다른 18개 파일은 안전합니다. 즉 1.3.4 코드 자체는 *법령/시행령/시행규칙/별표/별지서식* 에는 무해하고, **매뉴얼 PDF 한 파일이 article-aware split 의 본질적 적합 대상이 아닙니다**.**

회귀의 본질:
- 1.3.2 (현 인덱스 baseline) → 매뉴얼 1057청크 / 146 unique article
- 1.3.4 (시뮬) → 매뉴얼 857청크 / **46 unique article** / forced_split 비율 **94.7 %** / `제2조 → 310 parts`, `제65조 → 218 parts`

매뉴얼 PDF 문서 구조 분석 결과:
- 매뉴얼은 운영 가이드이지 법령 본문이 아님. `제N조(...)` 토큰은 매뉴얼 본문 *전체* 에서 442회 등장하지만 그중 **인용 포인터**(들여쓰기 10~30칸 표 셀, 본문 내 인라인 인용)가 **357건**, 진짜 *조문 헤더처럼 보이는* 0칸 들여쓰기 케이스가 84건.
- 84건 중에서도 *진짜 article 본문 시작* 은 부록 협약서 표준안(p353~359, ~35건) 정도이고, 나머지 49건은 표 셀 시작점 또는 인라인 인용. 즉 article-aware split 으로는 매뉴얼을 깨끗이 자르는 것이 불가능.
- 1.3.4 가 표 셀 인용 ≥10칸 들여쓰기를 차단했지만, *0칸 들여쓰기 표 셀 시작* 은 여전히 통과 → 그 결과 *하나의 거대한 article* 이 길이 분할만으로 잘게 쪼개져 part 수 폭증.

**권고**: chunker 1.3.4 추가 패치(자동 적용 금지) 또는 매뉴얼 PDF 의 라우팅 변경(article-aware 비활성, 페이지/문단 기반 청킹) 중 하나를 사용자 결정에 맡김. 자세한 내용은 §5.

법령 본체(혁신법/시행령/시행규칙) HWP 3개는 모두 HWPML 포맷이라 hwp-mcp가 지원하지 못해 **현재 인덱스에 빠져 있습니다**. 1.3.4 회귀와 *별개의 큰 문제*. §6 권고 추가.

## 5. 발견된 회귀 위험 + 권고 (자동 적용 금지 — 사용자 결정 대기)

### A. 가장 권장 — 매뉴얼 PDF 라우팅 분기 (article-aware split 비활성화)

매뉴얼은 운영 가이드이지 법령이 아니므로 article 단위 split 자체가 부적합합니다. 다음 두 옵션 중 하나를 추가:

**옵션 A1 (보수적)** — `chunk_document` 진입부에서 doc_type=가이드 또는 매뉴얼 파일명 매칭 시 article-aware split 을 건너뛰고 *페이지+문단 기반* 단순 청킹으로 라우팅. 청크 단위는 1500~2000자 윈도우, 페이지 경계 우선 break. article_no 는 `"가이드"` 또는 `f"p{page}"` 사용.
- 장점: 페이지 매핑이 정확. 검색 정확도는 BM25/리랭킹이 보강.
- 단점: 매뉴얼 안에서 "혁신법 제13조" 같은 query 가 article_no metadata 검색으로는 안 잡히고 본문 매칭에 의존.

**옵션 A2 (적극적)** — 매뉴얼 PDF 에 *heading-aware splitter* 추가. pdfplumber 의 polygon/font-size 정보로 헤딩 추출 → "제1장", "제2장 …", "Q&A" 등 매뉴얼 자체 목차 구조로 split.
- 장점: 매뉴얼 본문 자체 구조 따라감.
- 단점: 구현 시간 추가. 매뉴얼별로 헤딩 패턴이 달라 fragile.

### B. 차선 — chunker 1.3.5 패치 (article 매칭 추가 필터)

표 셀 0칸 들여쓰기 false positive 를 추가로 차단하려면:

```python
# 1.3.5 후보 필터:
# 1) 매칭된 article 헤더 라인의 *직후 줄* 이 같은 들여쓰기/포맷이면 표 셀로 추정
# 2) 한 페이지에 article 매칭이 ≥3개면 매핑표로 추정 → 그 페이지 통째로 차단
# 3) 매칭 라인 길이가 짧고 (~30자 미만) `(...)` 직후 본문이 없으면 표 셀
```

옵션 B 구현 시 매뉴얼은 깨끗해지지만 *진짜 부록 표준 협약서* (p353~359) 도 함께 차단됨 → 협약서 article_no 검색은 깨짐. 트레이드오프 명확.

### C. 매뉴얼 PDF 자체 재처리 — 표 셀 추출 모드 변경

`pdf_parser.py:135` 의 `page.extract_text(layout=True)` + `_extract_tables_as_text` 가 표 셀 내용을 *줄별로* 출력하면서 매뉴얼 매핑표가 각 셀을 0~30칸 들여쓰기로 펼침. 이걸 `extract_text(layout=False)` 로 바꾸거나, 표 페이지(p15~20)는 표 추출만 사용 (extract_text 무시) 로 바꾸면 매핑표 false positive 가 완화될 가능성. 다만 이는 다른 페이지에도 영향이 가므로 *큰 회귀 위험*.

### 결정 요청

세 옵션 중 어떤 길로 가야 할지 사용자 결정 필요. 현재 1.3.4 그대로 sync 하면 매뉴얼 PDF 결과는 **baseline 보다 *나빠진* 상태로 인덱싱** 됩니다 (제2조 310 parts, forced_split 94.7%). chunker 추가 패치 없이 sync 진행 권하지 않습니다.

## 6. HWPML 본체 3개 — 별도 이슈

법령 본체 (혁신법 / 시행령 / 시행규칙) HWP 3개 모두 HWPML(XML) 포맷 → hwp-mcp 0.1.x (OLE2 only) 미지원 → 인덱스 비어있음. 1.3.4 와 무관.


권고:
1. 사용자가 한컴오피스에서 PDF 변환 후 PDF 인덱싱 (가장 빠른 우회)
2. 또는 별도 HWPML XML 파서 추가 (`pyhwp` / `hwp5txt` / 직접 XML 파싱). HWPML 은 결국 XML 이라 직접 파싱 가능.
3. 또는 법제처 MCP 검색을 답변 시 hot-fetch 로만 사용 (인덱스 없이도 본문 인용 가능).
