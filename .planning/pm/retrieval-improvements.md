## 1. 한계별 분석

### 한계 1 — 인덱싱 누락 페이지 영구 검색 불가

**현재 영향**
- 사용자 시나리오: 매뉴얼 105p에 별표 형태의 비목 분류표가 스캔 이미지로만 들어 있는 경우. pdfplumber `extract_text`는 빈 문자열, OCR(`_ocr_page`)이 30자 미만 반환 → `ocr_flagged_pages`에 기록만 되고 인덱스에는 안 들어감. 이후 "비목 분류 알려줘" 질의에 응답 못 함. 더 위험한 시나리오: 사용자가 "이 매뉴얼에 그런 내용 없어요"로 오해 → 답이 없는데 있는 줄 모름.
- 현재 약 21페이지가 이 상태로 추정됨.

**근본 원인**
- `pipeline/pdf_parser.py:96-110` `_ocr_page` — tesseract 단일 백엔드. 한국어 표/저해상도/배경 컬러 페이지 인식률이 낮음.
- `pipeline/pdf_parser.py:144-156` — OCR 실패 시 단순히 경고만 출력하고 `pages`에 빈 텍스트로 추가 → 다운스트림에서 사용자에게 노출되지 않음.

**개선 옵션**

| # | 옵션 | 난이도 | 효과 | 외부 의존성 | 트레이드오프 |
|---|---|---|---|---|---|
| 1A | OCR 전처리 강화 (DPI 상향 + Pillow로 그레이스케일·이진화·콘트라스트) | 하 | 중 (10~15p 추가 복구 예상) | 없음 (Pillow 기설치) | 처리 시간 1.5~2배. 표 셀이 점선이면 여전히 실패. |
| 1B | OCR 백엔드 다중화 — tesseract 실패 시 PaddleOCR(Korean) 폴백 | 중 | 상 (한국어 표/세로쓰기 대응 우수) | `paddleocr`, `paddlepaddle` 추가 (수백 MB) | 디스크/메모리 비용 큼. macOS arm64 휠 안정성 점검 필요. |
| 1C | 실패 페이지를 "공백 청크"로 인덱싱 + UI에서 "이 페이지는 OCR 실패" 안내 | 하 | 낮~중 (검색은 못하지만 누락 사실 사용자에게 노출) | 없음 | 검색 품질 자체는 안 오름. 그러나 "근거 없음" 원칙과 정합. |
| 1D | 외부 OCR API (Naver CLOVA OCR / Upstage Document AI) 호출 | 중 | 상 (한국어 정부문서 SOTA) | 유료 API 키, 네트워크 | 로컬 RAG 원칙 일부 훼손, 비용 발생, 1회성 인덱싱이라 비용은 작음. |

**추천**: **1A → 1C 병행** (1B, 1D는 보류).
- 이유: 1A는 의존성 추가 없이 즉시 효과. 1C는 "근거 없으면 답 안 한다" 원칙과 직결되며 사용자 신뢰 확보. 1B/1D는 ROI 대비 운영 부담이 커서 21p 잔여를 위한 투자로 과함. 잔여가 5p 미만으로 줄지 않으면 그때 1B 재검토.

### 한계 2 — 청크 경계가 의미를 자르면 검색 실패

**현재 영향**
- 사용자 시나리오: "제15조 제2항 제3호의 단서를 알려줘" → `_split_long_text`가 항/호 마커에서 분할하면서 단서(다만 ~) 부분만 별도 청크로 분리되면, 임베딩이 단서 의미를 상위 조항과 단절시켜 코사인 유사도가 떨어짐. retriever top-8에서 누락 → 답변 불가능.
- 또 다른 시나리오: 조문 끝에 "이 조의 적용은 다음 각 호의 사유에 한한다" 같은 후행 문장이 다음 조문 헤더와 함께 묶이면서 잘못된 article_no 라벨이 붙음.

**근본 원인**
- `pipeline/chunker.py:96-139` `_split_by_articles` — 하드 분할. 조문 간 cross-reference는 끊김.
- `pipeline/chunker.py:197-214` `_split_long_text` — 의미 단위 분할 후 `_greedy_pack`으로 다시 합치지만, 항/호 경계는 무시. (part 1/3) 식 라벨만 부여되어 사용자가 어떤 항인지 모름.
- 페이지 경계가 조문 중간을 가르면 `_offset_to_page`가 첫 글자 페이지만 기록 → 후반 항이 다른 페이지인데 동일 page 메타로 들어감.

**개선 옵션**

| # | 옵션 | 난이도 | 효과 | 외부 의존성 | 트레이드오프 |
|---|---|---|---|---|---|
| 2A | 슬라이딩 윈도우 오버랩 (예: 200자 stride) — 인접 청크 경계 텍스트 공유 | 하 | 중 | 없음 | 인덱스 크기 ~30% 증가. 중복 매칭으로 retriever top-k 다양성 저하 가능. |
| 2B | 계층적 청킹 (조 → 항 → 호) + 자식 청크에 부모 조문 헤더 prefix 포함 | 중 | 상 | 없음 | 구현 복잡. ChunkMetadata에 parent_id 필드 추가 필요. |
| 2C | Parent-doc retrieval — 작은 청크(항)로 검색하고, 답변 단계에서 부모 조 전문을 컨텍스트로 주입 | 중 | 상 | 없음 (Qdrant payload에 parent_text 저장) | 컨텍스트 길이 증가 → 답변 모델 토큰 비용 증가. answerer 변경 필요. |
| 2D | LLM 기반 의미 청킹 (Anthropic API로 청크 경계 결정) | 상 | 상 | API 비용, 인덱싱 시간 ↑↑ | 비결정적. 재인덱싱 비용 큼. |

**추천**: **2B (계층적 청킹) + 2C (parent-doc retrieval) 결합**.
- 이유: 한국 법령은 조-항-호 위계가 명확하고 사용자가 항/호 단위로 질의함. 2B로 검색 정확도, 2C로 답변 시 누락된 단서/예외 조항 보완. 2A는 손쉬우나 항/호 의미가 깨진 채 텍스트만 겹쳐서 근본 해결 안 됨. 2D는 ROI 부족.
- 페이지 매핑 버그(`_offset_to_page` 첫 글자만 기록)는 2B 작업과 함께 수정.

### 한계 3 — 표/별표가 통째 묶여 검색 정확도 낮음

**현재 영향**
- 사용자 시나리오: "별표 2 직접비 항목 중 외주용역비 한도?" → 별표 2가 5만 자라면 `_split_long_text`가 줄 단위 그리디 패킹으로 ~25개 청크로 자름. 각 청크는 표 헤더/맥락 없이 행 일부만 포함 → "외주용역비"가 들어간 행이 1개 청크에 있어도 "한도 50%" 같은 비율 행은 다른 청크에 있어서 답변 단계에서 둘을 동시에 못 잡음.
- ko-sroberta는 max 512 token (≈한국어 700~900자) — 청크가 1000자 초과면 뒷부분이 임베딩에 반영 안 됨. `MAX_CHUNK_LEN=2000`은 임베딩 모델 한계를 초과함 (`pipeline/chunker.py:29`).

**근본 원인**
- `pipeline/chunker.py:29` `MAX_CHUNK_LEN = 2000` — 임베딩 모델 컨텍스트 초과.
- `pipeline/chunker.py:184-194` `_split_by_lines` — 표를 일반 텍스트로 취급. 헤더 행 보존 안 됨.
- `pipeline/pdf_parser.py:41-55` `_extract_tables_as_text` — 표를 `" | "` 조인으로 평탄화. 셀 위계 (그룹 헤더 등) 손실.

**개선 옵션**

| # | 옵션 | 난이도 | 효과 | 외부 의존성 | 트레이드오프 |
|---|---|---|---|---|---|
| 3A | `MAX_CHUNK_LEN`을 700자로 낮추고, 별표는 행 단위로 분할 + 표 제목+헤더를 모든 행 청크 prefix에 추가 (contextual prefix) | 중 | 상 | 없음 | 청크 수 4~5배 증가. 인덱스 디스크 ↑. |
| 3B | 표 전용 별도 컬렉션 — `doc_type=별표`인 청크는 BM25/키워드 위주로만 검색 (벡터 점수 무시) | 중 | 중 | 없음 (한계 4 BM25와 결합) | 두 컬렉션 관리 부담. answerer 라우팅 로직 필요. |
| 3C | 표를 row-level dict로 파싱해 각 행을 "헤더A: 값A, 헤더B: 값B" 자연어 문장으로 변환 후 청크화 | 상 | 상 | 없음 (pdfplumber 기능 활용) | pdfplumber가 셀을 못 인식하는 표(병합셀, 라인 없는 표)는 실패. |
| 3D | Anthropic contextual retrieval 패턴 — 각 행 청크에 "이 행은 별표 X 직접비 분류표의 Y행이며 …" LLM 생성 요약 prefix | 상 | 상 | API 비용, 1회성 | 인덱싱 시간 큼. 결과는 매우 우수. |

**추천**: **3A 우선 적용, 3D는 별표만 선택적으로 시도**.
- 이유: 3A는 즉시 임베딩 모델 한계와 정합되고 의존성 없음. "외주용역비"가 들어간 행 청크가 별표 제목과 헤더를 함께 보유 → 검색 시 표 맥락 보존. 3C는 표 다양성으로 실패 케이스가 많아 1차 릴리스에서 위험. 3D는 ROI 좋으나 외부 API 호출 정책 확정 후로 미룸.

### 한계 4 — 순수 벡터 검색 → 정확 키워드("제15조") 약함

**현재 영향**
- 사용자 시나리오: "제15조 제3항을 그대로 보여줘" → 벡터 검색으로 "제15조"는 의미 임베딩이 약하므로 "연구개발비 사용", "비목" 같은 의미 유사 청크가 더 높은 점수를 받음. top-8에 제15조 청크가 누락. 사용자는 매뉴얼에 분명히 있는 조문을 못 받음.
- 또 다른 시나리오: 사용자가 문서 종류를 알고 있는데도 ("운영요령에 있는 거" → `doc_type` 사전 필터로 좁혀야 하나 retriever는 벡터 점수만으로 잘못된 doc_type 청크를 선택).
- 현재 가장 빈번한 실패 모드. 다른 한계를 다 고쳐도 retrieval이 청크를 못 꺼내면 의미 없음.

**근본 원인**
- `pipeline/retriever.py:41-47` — `client.query_points`에 vector만 전달. dense vector single-stage.
- `pipeline/retriever.py:36-39` — `doc_type` 필터는 받지만 호출자(answerer)가 사용 안 함.
- 청크 메타에 `article_no`가 있지만 인덱스(payload index)가 없어 fast filter 불가.

**개선 옵션**

| # | 옵션 | 난이도 | 효과 | 외부 의존성 | 트레이드오프 |
|---|---|---|---|---|---|
| 4A | 정규식 prefilter — 질의에서 "제\\d+조(의\\d+)?" 추출 → Qdrant payload filter `article_no` exact match → 매칭 시 단축 경로 직답 | 하 | 매우 상 (조문번호 질의 100% 해결) | 없음 (Qdrant 기능) | "제15조"가 질의에 없는 일반 질문에는 도움 안 됨. payload index 생성 필요. |
| 4B | BM25 + 벡터 하이브리드 (Reciprocal Rank Fusion) | 중 | 상 | `rank_bm25` 또는 `qdrant-client` BM25 sparse vector 지원 (1.10+, 현재 1.17) | 두 인덱스 운영. RRF 가중치 튜닝 필요. |
| 4C | Cross-encoder 리랭킹 (예: `bongsoo/kpf-cross-encoder` 또는 `dragonkue/bge-reranker-v2-m3-ko`) — 1차 top-30 → 리랭커로 top-8 | 중 | 상 | `sentence-transformers`로 로드 (의존성 추가 없음). 모델 다운로드 ~500MB. | 추론 지연 +200~500ms. |
| 4D | 메타필터 강화 — answerer가 질의에서 `doc_type` 힌트 ("운영요령", "법") 추출 후 retriever에 전달 | 하 | 중 | 없음 | 사용자 의도 추정 오류 시 결과 누락. |

**추천**: **4A + 4B 조합 (1차) → 4C 추가 (2차)**.
- 이유: 4A는 가장 빈번한 실패(조문번호 직접 질의)를 거의 완전히 해결. 구현 30~60분. 4B는 별표/매뉴얼 키워드 질의("외주용역비") 회수율 크게 개선. Qdrant 1.10+에서 sparse vector 네이티브 지원 → `rank_bm25` 외부 의존성 추가 불필요 가능성 (scout 위임 후보). 4C는 답변 정밀도 마지막 1할이며 4A/4B 효과 측정 후 도입.

### 한계 5 — 증분 인덱싱 부재

**현재 영향**
- 사용자 시나리오: 운영요령 개정판이 같은 파일명으로 덮어쓰기됨 → `ingest_done.txt`에 경로가 있으므로 스킵 → 구판이 인덱스에 그대로 남고 신판은 들어가지 않음. 답변에 폐지 조문이 인용됨.
- 사용자가 새 PDF를 `data/uploads/`에 떨어뜨려도 `python batch_ingest.py`를 수동 실행해야 함. UI(Streamlit)에서 즉시 반영 안 됨.
- `--force` 시 전체 재인덱싱이라 200+ PDF에서 30분~1시간 소요.

**근본 원인**
- `batch_ingest.py:23-26` `load_done` — 경로만 set으로 저장. 내용 변경 감지 불가.
- `batch_ingest.py:144` `upsert_chunks` — 기존 doc의 청크를 삭제하지 않음 → 재인덱싱 시 중복 누적 가능 (force 모드에서도).

**개선 옵션**

| # | 옵션 | 난이도 | 효과 | 외부 의존성 | 트레이드오프 |
|---|---|---|---|---|---|
| 5A | SHA256 해시 + mtime 기반 manifest (JSON: `{path: {hash, mtime, doc_id}}`) → 변경 감지 시 해당 doc_id의 청크 일괄 삭제 후 재인덱싱 | 중 | 상 | 없음 | 첫 마이그레이션 시 기존 `ingest_done.txt` 폐기 필요. |
| 5B | watchdog 데몬으로 디렉토리 감시 → 자동 인덱싱 큐 | 상 | 중 | `watchdog` 추가 | 백그라운드 프로세스 운영 부담. Streamlit과 별도 프로세스. |
| 5C | Streamlit 업로드 핸들러에서 단일 파일 즉시 인덱싱 호출 + UI 인디케이터 | 중 | 상 (UX 개선) | 없음 | Streamlit 세션 동안 인덱싱 실행 → 사용자 대기. 대용량 PDF는 느림. |

**추천**: **5A + 5C 결합**. 5B는 보류.
- 이유: 5A는 "내용 변경 시 재인덱싱" 정확성 핵심. 5C는 Streamlit UX 직결. 5B는 사용자 수가 적은 사내 도구에서 과한 인프라.

## 2. 한계 간 의존성

```
한계 5 (증분 인덱싱)
    ↓ 빠른 반복 가능 → 한계 1/2/3 시도 비용 ↓
한계 4 (하이브리드/리랭킹)  ← 선결조건 없음. 가장 독립적이며 즉시 효과
    ↓ 검색이 청크를 꺼낼 수 있어야
한계 2/3 (청킹) — 검색이 동작하지 않으면 청킹 개선 효과 측정 불가
    ↓
한계 1 (OCR) — 가장 marginal. 21p tail.
```

**핵심 의존성**
- **한계 4 → 한계 2/3**: 한계 2/3을 먼저 고치면, 회귀 측정 수단(=하이브리드 검색)이 없어 효과 검증 불가. 따라서 **한계 4를 먼저** 처리해야 한계 2/3 작업의 ROI를 정량화할 수 있음.
- **한계 5 → 한계 1/2/3**: 청킹/OCR 시도마다 전체 재인덱싱(30분+)이면 반복 비용이 폭증. 한계 5를 먼저 두면 변경 doc만 재인덱싱 가능 → 시도-측정 사이클이 5분 이내로 단축.
- **한계 4의 4A는 단독**: 정규식 prefilter는 청킹/OCR과 독립. 하이브리드/리랭커보다 먼저 30분 안에 가능.
- **한계 1은 후순위**: 다른 모든 한계의 회수율 개선이 잔여 21p OCR보다 효과 큼.

## 3. 페이즈 로드맵 (3단계)

### Phase A — Retrieval Foundation (한계 4 + 평가셋 준비)

- **목표**: 검색 실패 모드 측정·해결의 기반 확보. 조문번호 직접 질의 ≥95% 회수.
- **다루는 한계**: 4 (전부), 추가로 평가셋 30~50문항 골드 정답 구축.
- **예상 작업량**: 2~3일 (16~24h).
- **리스크**: 중. Qdrant sparse vector 네이티브 지원 여부 확인 필요. 미지원 시 `rank_bm25` 외부 추가 정당화.
- **scout 위임 필요**: 예. 토픽 = "Qdrant 1.17에서 sparse vector(BM25) 네이티브 지원 여부와 `Prefetch` API 사용법 + 한국어 cross-encoder 리랭커 추천 (HuggingFace 무료 모델, 한국어 법률 도메인)".
- **backend 작업 범위**:
  - `pipeline/retriever.py` — 정규식 prefilter (4A), 하이브리드 검색 (4B), 리랭킹 (4C 옵션).
  - `pipeline/indexer.py` — payload index 생성 (article_no, doc_type, doc_name).
  - `eval/` 신설 — 30~50문항 골드셋 (조문번호 질의 10, 키워드 질의 10, 표/별표 10, 일반 질의 10). recall@8, MRR 측정 스크립트.
  - `pipeline/answerer.py` — doc_type 힌트 추출 → retriever 전달 (4D).
- **완료 기준 (acceptance criteria)**:
  - 조문번호 질의 골드셋 recall@1 ≥ 95%, recall@8 = 100%.
  - 전체 골드셋 recall@8 ≥ 80% (현재 베이스라인 측정 후 +30%p 목표).
  - `eval/run.py` 실행으로 회귀 측정 가능.
  - 기존 인덱스로 동작 (재인덱싱 불필요).

---

### Phase B — Chunking & Ingest Hygiene (한계 2, 3, 5)

- **목표**: 청크 의미 단위 정렬 + 콘텐츠 변경 자동 감지.
- **다루는 한계**: 2 (계층적 청킹 + parent-doc), 3 (`MAX_CHUNK_LEN` 조정 + 별표 contextual prefix), 5 (해시 manifest + Streamlit 단일 업로드).
- **예상 작업량**: 3~4일 (24~32h).
- **리스크**: 중. 재인덱싱 1회 필요 (전체 200+ PDF, 30~60분). 청킹 변경으로 Phase A 골드셋 점수 변동 → 회귀 비교 필수.
- **scout 위임 필요**: 가능. 토픽 = "한국 법령 문서(법률/시행령/별표) RAG 청킹 베스트 프랙티스 — Anthropic contextual retrieval, parent-doc retriever 패턴 LangChain 구현 사례, 별표/표 청킹 전략".
- **backend 작업 범위**:
  - `pipeline/chunker.py` — `MAX_CHUNK_LEN=700` 조정, 계층적 청킹 (parent_id, parent_text 필드 추가), 별표 contextual prefix.
  - `pipeline/chunker.py:69-77` `_offset_to_page` — 청크 후반 페이지도 정확 매핑 (range 보유).
  - `pipeline/answerer.py` — 자식 청크 검색 → 부모 조 전문 컨텍스트 주입 옵션.
  - `batch_ingest.py:20-32` — manifest JSON 도입, 해시 기반 변경 감지, 변경 doc만 재인덱싱.
  - `pipeline/indexer.py` — `delete_by_doc_name` 헬퍼 (재인덱싱 전 청크 삭제).
  - `app.py` — 업로드 시 단일 파일 인덱싱 진행률 표시.
- **완료 기준**:
  - Phase A 골드셋 recall@8이 Phase A 결과 대비 동일 또는 상승. 하락 시 롤백.
  - 별표 행 단위 질의 ("외주용역비 한도") recall@8 ≥ 80%.
  - 같은 파일 내용만 변경 후 재인덱싱: 변경 doc 청크만 삭제·재생성, 무관 doc 변동 0.
  - Streamlit 업로드 → 30초 이내 검색 가능.

---

### Phase C — Robustness Tail (한계 1)

- **목표**: OCR 실패 페이지 잔여 ≤5p, 실패 페이지의 사용자 가시화.
- **다루는 한계**: 1.
- **예상 작업량**: 1~2일 (8~16h).
- **리스크**: 낮. 격리된 변경. 임베딩/검색에 영향 없음.
- **scout 위임 필요**: 선택. 토픽 = "한국어 PDF OCR — Pillow 전처리(이진화, 노이즈 제거) tesseract 정확도 향상 베스트 프랙티스".
- **backend 작업 범위**:
  - `pipeline/pdf_parser.py:96-110` — `_ocr_page`에 Pillow 전처리 파이프라인 (그레이스케일 → 이진화 → 노이즈 제거 → 콘트라스트). DPI 옵션 400으로 상향(현재 300).
  - `pipeline/pdf_parser.py:144-156` — OCR 실패 페이지를 placeholder 청크로 인덱싱 (text="[OCR 실패 — 본 페이지는 검색 불가]", page metadata 보유).
  - `app.py` — 답변 컨텍스트에 placeholder 청크가 포함되면 사용자에게 "p.NN OCR 실패 안내" 노출.
- **완료 기준**:
  - 현재 21p 잔여가 ≤5p로 감소.
  - 실패 페이지를 사용자가 UI에서 인지 가능 (silent skip 제거).
  - Phase A 골드셋 점수 변동 없음.

## 4. 첫 번째 액션 (단일 추천)

**Phase A의 4A — 정규식 prefilter를 `pipeline/retriever.py`에 추가**.

**왜 이것이 첫 번째인가**
1. **빈도 가중 영향이 가장 큼**. 사용자 질의 중 "제N조" 패턴 직접 질의가 30~40% 추정. 현재 vector-only로는 이 부류가 가장 자주 실패.
2. **30~60분 안에 완료**. 정규식 추출 + Qdrant payload filter 조합. 외부 의존성 0.
3. **기존 인덱스로 동작**. 재인덱싱 불필요. Streamlit 베이스라인 그대로 즉시 효과 확인.
4. **회귀 측정의 닻이 됨**. 4A 완료 후 골드셋 30문항(특히 조문번호 10문항)을 만들면, 이후 4B/4C/Phase B/C 각 변경의 효과를 정량적으로 비교 가능.
5. **실패해도 손실 작음**. 격리된 함수, 기존 search 경로는 fallback으로 유지.

**즉시 후속**: 4A 완료 → eval 골드셋 30문항 작성 → 베이스라인 점수 기록 → Phase A 4B/4C 진행.

**위임 결정**:
- 4A 자체는 backend에 직접 위임 (scope 작음).
- Phase A 시작 전 scout에 "Qdrant 1.17 sparse vector 지원 + 한국어 cross-encoder 무료 모델" 리서치 1건 병행 의뢰. 결과가 4B/4C 의존성 결정에 영향.
